远程调用没那么难：深入浅出gRPC

0xccio
2025-09-21
gRPC

前段时间我突然发现，自己对 gRPC 的理解其实挺尴尬的 🤔。
平时写 Go 项目的时候经常用到 gRPC，服务端、客户端都照着 Demo 跑得挺顺利，但如果有人突然问我：

“gRPC 到底是怎么让你在 Go 里像本地函数一样去调远程服务的？”

我估计只能支支吾吾地说出几个关键词：protobuf、HTTP/2、快！
但这显然不够。所以这次写下这篇文章，梳理一下 RPC 的核心思想，以及 gRPC 在 Go 项目里到底该怎么用。

1. RPC 的本质是什么？

RPC（Remote Procedure Call）翻译过来就是“远程过程调用”。听起来很高大上，其实一句话：

RPC 要让你调用远程机器上的函数，感觉就像调用本地函数一样简单。

它背后的关键就是一个“障眼法”：

  1. 在客户端，你调用的并不是“远端函数”，而是一个 代理函数（Stub）。
  2. 代理函数帮你把参数打包成字节（序列化），通过网络送到服务端。
  3. 服务端再解包（反序列化），调用真正的业务函数。
  4. 结果再打包，传回客户端。

2. gRPC: Google出品高配版

2.1 gRPC 的总体架构

gRPC 本质上是一个 四层组合拳：

  1. 接口定义层（IDL）

    • 用 .proto 文件定义服务、方法、参数、返回值。

    • 这是跨语言的契约，可以自动编译成不同语言的 Stub（代理代码）。

  2. 序列化层（Protobuf）

    • 把对象（结构体）转成紧凑的二进制流（序列化），再传输。

    • 接收端再解包（反序列化）恢复成对象。

  3. 传输层（HTTP/2）

    • 所有 RPC 调用都跑在一条 HTTP/2 连接上，支持多路复用。

    • 天生支持流式调用（单向、多向、双向）。

  4. 运行时（gRPC Runtime）

    • 提供拦截器、负载均衡、认证、状态码模型、健康检查等“增强功能”。

    • 开发者只管写业务逻辑，剩下的交给 gRPC runtime 管。

2.2 gRPC 是如何一步步完成 RPC 调用的？

  1. 调用本地函数（看似）

    • 我们写的代码只是：resp, err := client.Hello(ctx, req)

    • 实际上调用的是 Client Stub（代理）。

  2. 序列化请求

    • Stub 会帮我们把 req 转成 Protobuf 二进制。

    • 比如一个 {name:"gopher"} 会变成压缩过的字节流。

  3. 通过 HTTP/2 发出去

    • gRPC 在底层维护一条长连接。

    • 它会在 HTTP/2 里新建一个 stream，把请求包发过去。

    • 同时附带 metadata（比如 trace-id、token）。

  4. 服务端解码 + 分发

    • 服务端 gRPC runtime 收到包，找到对应的 Server Stub。

    • Server Stub 反序列化字节流，恢复成 HelloRequest 对象。

    • 然后调用服务端这边的业务函数 Hello(ctx, req)。

  5. 执行函数并返回结果

    • 业务函数跑完，返回 HelloReply。

    • Server Stub 序列化结果，写回 HTTP/2 stream。

  6. 客户端解码并返回

    • Client Stub 收到响应字节流，解码为 HelloReply。

    • 你的业务代码拿到结果，就像在调用本地函数一样。

3. gRPC怎么用？

在了解底层原理后，当然还得知道我们怎么在项目中去集成gRPC了

我自己总结了个小口诀：

订契约(proto)→ 生代码(stub)→ 写业务(handler)→ 起服务(server)/拨号(client)

其实这就是我们需要掌握的全部东西了，接下来就走进其中再仔细看看。

3.2 生代码

前面我们写好了 .proto 文件，这个文件就是契约（IDL）。但光有契约，Go 项目还没法直接用。 这时候就需要借助 protoc 和 插件，把 .proto 翻译成我们在 Go 项目里能直接 import 的代码。

1.安装必要工具

gRPC 的代码生成需要 protoc 和两个插件：

  • protoc：Protobuf 的编译器

  • protoc-gen-go：生成普通的 Go 结构体（message）

  • protoc-gen-go-grpc：生成 gRPC 专用的客户端、服务端接口（service）

解释一下：

  • -I . → 指定 proto 文件的根目录

    • 它告诉 protoc 去哪里找 .proto 文件。

    • 当你在一个 proto 文件里用 import "xxx.proto"; 的时候，protoc 就会从 -I 指定的路径里去查找。

    • 使用"."就是以当前目录作为 proto 的根目录。

  • --go_out=. → 把 message 相关的 Go 代码生成到当前目录

  • --go-grpc_out=. → 把 gRPC 相关的 Go 代码生成到当前目录

  • --go_opt=paths=source_relative → 保证生成的 .pb.go 和 .proto 在同一目录，方便管理

  • 最后是 proto 文件路径
